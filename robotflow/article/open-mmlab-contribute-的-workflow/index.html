open-mmlab contribute 的 workflow

提交 PR 前，contributor-side 的准备工作

参考资料：https://github.com/open-mmlab/mmdetection/blob/master/.github/CONTRIBUTING.md

What kind of contributions?

All kinds of contributions are welcome, including but not limited to the following.

  • Fixes (typo, bugs)
  • New features and components

Workflow

  • fork and pull the latest mmdetection
  • checkout a new branch (do not use master branch for PRs)
  • commit your changes
  • create a PR

Note

  • If you plan to add some new features that involve large changes, it is encouraged to open an issue for discussion first.

代码风格

和大多数开源项目一样，mmdetection 有着非常严格的代码规范。

在 PEP8 标准的统领下，mmdetection 采用以下工具来检查代码风格：

  • flake8: linter
  • yapf: formatter
  • isort: sort imports

在参考资料中，提到了一个 pre-commit hook：https://github.com/open-mmlab/mmdetection/blob/master/.github/CONTRIBUTING.md。它可以在每次 commit 前都检查一遍所 stage 的文件的代码风格，非常方便，推荐使用。

正式提交 PR 后的工作流

代码的 License

mmdetection 使用的 License 是阿帕奇2.0，提交 PR 后就必须在 @CLAassistant 的引导下签署 CLA。主要内容就是认可 mmdetection 的 License，并且本人保证不含有未授权的代码。

Review

签署完成 CLA 后，open-mmlab 方面就会组织代码 review。review 的主要内容更多是代码风格上的，包括注释、变量名规范等；还有架构层面上的，比如会要求 contributor 使用框架中已有的某个模块替换掉自己写的某个模块，或者修改模块的 input / output，使新的模块可以和已有的模块缝合起来。

Conflicts

由于 Review 的环节可能比较长，在 review 环节结束后可能需要重新 merge 最新的 master 分支代码以解决 conflicts。

Unit-test

mmdetection 采用了一些自动化工具来测试代码、检查代码风格。（build-bot）

所有代码都必须通过这个自动化测试。

Merge

在以上环节结束后，open-mmlab 方面就会 merge 代码。
